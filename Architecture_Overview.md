# aut0arch System Architecture Overview

The `aut0arch` system is an automated architecture discovery and explanation pipeline. It accepts a Java repository URL, parses the source code into a semantic dependency graph, groups elements into higher-level architectural components, and leverages a Large Language Model (Gemini) to explain each component's purpose. A React frontend then visualizes this recovered architecture.

Here is a breakdown of the 5 repositories/components in the system and how they work together:

## 1. `server` (The Pipeline Orchestrator)
**Tech Stack:** Python, Flask
**Role:** The main entry point that ties the entire pipeline together. It exposes a `/analyze` REST endpoint that accepts a GitHub repository URL. The `orchestrator.py` script clones the target repository into a temporary directory and sequentially executes the `parser`, `clustering`, and `explainer` tools to generate the final architecture data.

## 2. `parser` (The Code Analyzer)
**Tech Stack:** Python, Tree-sitter
**Role:** Extracts the low-level structure of the target Java code. It uses `tree-sitter-java` to perform a strict parsing of the code, identifying files, classes, interfaces, and methods. It performs a 2-pass strategy to discover symbols and link method calls to their definitions across files. The final granular dependency graph is exported as `graph.json`.

## 3. `clustering` (The Architecture Abstractor)
**Tech Stack:** Python
**Role:** Takes the highly granular dependency graph (`graph.json`) generated by the parser and groups related nodes into cohesive higher-level modules or components. It outputs a `clustered_structure.json` file representing the abstracted architecture of the system.

## 4. `explainer` (The LLM Summarizer)
**Tech Stack:** Python, Google Gemini API
**Role:** Adds semantic meaning to the abstracted architecture. It reads the `clustered_structure.json` and the corresponding source code for each cluster, then prompts the `gemini-flash-latest` model to generate a high-level description for each component. The LLM identifies the role, likely inputs/outputs, technical details, and key patterns. The final enriched data is saved as `clustering_data.json` for the frontend to consume.

## 5. `app` (The Frontend Visualizer)
**Tech Stack:** React, Vite, Cytoscape.js
**Role:** The web-based user interface for the system. It uses `react-cytoscapejs` and `cytoscape-dagre` to visually render the architectural nodes and their flow/dependencies. Users can interact with the graph to view the LLM-generated explanations, technical details, and code evidence for each recovered component.

---
### Data Flow Pipeline
1. `server` clones Repo → `/temp_repos/`
2. `parser` reads `/temp_repos/` → outputs `parser/graph.json`
3. `clustering` reads `parser/graph.json` → outputs `clustering/clustered_structure.json`
4. `explainer` reads `clustering/clustered_structure.json` & source code → outputs `app/public/clustering_data.json`
5. `app` serves the visualization reading from `app/public/clustering_data.json`
